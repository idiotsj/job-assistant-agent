# 前端审查文档（2026-05-04）

本文档用于记录本轮 `apps/web` 前端改动的结构性审查结论，重点关注：

*   是否与当前设计规范冲突
*   是否会影响后续多人维护
*   是否会造成全局布局、导航或时间语义回归
*   是否会误导后续接手同学对当前实现状态的判断

本次审查对象主要包括：

*   `apps/web/src/app/layout.tsx`
*   `apps/web/src/app/globals.css`
*   `apps/web/src/components/layout/*`
*   `apps/web/src/components/blocks/*`
*   `apps/web/docs/frontend-design-spec.md`

---

## 1. 审查结论摘要

本轮改动已经开始把项目从旧版单侧栏工作台切向新的品牌顶栏与仪表盘壳层，这个方向本身是对的，也和设计文档的阶段规划一致。

但当前实现里有几处问题已经达到需要尽快处理的级别：

1. 根布局把 `DashboardShell` 强行挂到了所有路由上，破坏了双壳层路线。
2. 全局知识树暴露了还不存在的 `/interview` 路由，形成主导航 404。
3. 设计文档的“当前实现状态”没有同步更新，已经和当前分支不一致。
4. 周视图日期生成存在时区敏感问题，后续接真实数据时容易出现日期偏移。

整体判断：

*   **方向正确，但当前不建议把这版视为“壳层已经稳定可继续全面铺开”的基线。**
*   建议先修复 `P1` 问题，再让更多页面切到新壳层。

## 2. Findings

### [P1] 根布局把仪表盘壳层强制应用到了所有页面

**位置**

*   [layout.tsx](/D:/code/work%20agent/apps/web/src/app/layout.tsx#L20)

**问题说明**

当前 `RootLayout` 直接在应用根层挂载了：

*   `DashboardShell`
*   左侧 `CountdownCard`
*   右侧 `KnowledgeTreeCard`
*   右侧 `GradientAdviceCard`

这意味着所有页面都会被包进同一套三栏仪表盘结构，包括：

*   `/login`
*   `/register`
*   后续应当走 `ConversationShell` 的 `/advisor-chat`
*   未来任何需要独立布局的页面

这与设计文档中已经明确写死的“双壳层体系”冲突，也会导致后续为了做对话页或独立认证页，只能出现：

*   在仪表盘壳层内再套一层对话壳层
*   在仪表盘壳层内硬做一个看起来不像仪表盘的页面

这两种都不是正确方向。

**影响**

*   破坏页面布局分层边界。
*   后续 route group 难以自然分配不同壳层。
*   登录页 / 注册页会继续背着不必要的全局侧栏内容。
*   后续 `/advisor-chat` 很容易出现壳层嵌套错误。

**建议处理**

建议把壳层选择从根布局移到 route group 级别，例如：

*   根布局只保留：
    *   `html`
    *   `body`
    *   `AuthProvider`
    *   全局样式
*   仪表盘页面单独使用一个 route group layout
*   对话页面单独使用一个 route group layout
*   认证页面使用最轻布局或独立 auth layout

**建议优先级**

*   必须在继续大面积页面迁移前处理。

### [P1] 全局知识树现在会把用户导航到不存在的 `/interview`

**位置**

*   [knowledge-tree-card.tsx](/D:/code/work%20agent/apps/web/src/components/blocks/knowledge-tree-card.tsx#L16)

**问题说明**

`KnowledgeTreeCard` 默认链接中包含：

*   `面试模拟 -> /interview`

但当前仓库的 `src/app` 下还没有 `/interview` 路由。

由于 `KnowledgeTreeCard` 现在又被挂到了根布局右侧栏上，这个死链接不再只是某个局部页面的问题，而是：

*   几乎所有页面都会展示这个入口
*   用户会把它当成正式主导航入口
*   点击后直接进入 404

**影响**

*   主导航可信度下降。
*   评审或联调时容易误判“面试模拟页已经有入口，应该只是页面没写完”。
*   后续页面未完成时如果继续照这个模式加入口，会形成更多全局坏链。

**建议处理**

在对应路由落地前，二选一：

1. 从默认知识树里移除该入口。
2. 临时指向一个明确的占位页，但页面必须显式说明“模块建设中”。

更推荐第一种，因为当前项目文档已经明确“未完成模块不要伪装成已落地能力”。

### [P2] 设计文档的实现状态已经与当前分支不一致

**位置**

*   [frontend-design-spec.md](/D:/code/work%20agent/apps/web/docs/frontend-design-spec.md#L1474)

**问题说明**

设计文档第 7 节仍写着：

*   顶部品牌与身份条未正式落地
*   右侧综合就业知识树未正式落地
*   双壳层体系未落地

但当前分支代码里已经：

*   新建并使用了 `BrandTopBar`
*   在根布局挂入了 `KnowledgeTreeCard`
*   在根布局挂入了 `DashboardShell`

也就是说，代码和文档现在至少已经部分分叉。

**影响**

*   后续接手同学会被错误基线误导。
*   评审时会出现“代码说已经上了，文档说还没上”的双重口径。
*   影响下一轮任务拆分，因为别人不知道哪些东西是“未开始”，哪些其实只是“实现方式还需要调整”。

**建议处理**

不要简单把状态改成“已完成”，而应更新成更准确的描述，例如：

*   顶部品牌条与新仪表盘壳层已开始接入
*   当前仍是全局硬挂阶段，尚未完成 route-group 级壳层分配
*   右侧知识树已作为共享块出现，但导航策略仍未稳定

这样既能反映已有进展，也不会掩盖当前实现仍有结构问题。

### [P2] 周视图日程板日期生成存在时区偏差风险

**位置**

*   [weekly-schedule-board.tsx](/D:/code/work%20agent/apps/web/src/components/blocks/weekly-schedule-board.tsx#L21)

**问题说明**

当前周视图使用：

*   `d.toISOString().slice(0, 10)`

来生成日期字符串。

这个写法会先把本地时间转成 UTC，再截取日期部分。在 `Asia/Shanghai` 这类正时区下，午夜附近的本地时间可能会被转成前一天的 UTC 日期，导致：

*   `date` 字段偏移一天
*   后续 `new Date(day.date).getDate()` 的结果也跟着偏

现在这个组件主要还是演示块，问题还比较隐蔽；但一旦后续把真实日程或个人中心周视图接进来，这个偏移会变成真实 UI 错误。

**影响**

*   周几与日期数字可能不一致。
*   与真实 schedule 数据对接时容易出现错误定位困难的问题。
*   时区问题会表现为“偶发”，更难排查。

**建议处理**

应使用本地日期组装方式，而不是 `toISOString()`，例如用本地：

*   `year`
*   `month`
*   `day`

拼出 `YYYY-MM-DD` 字符串。

## 3. 建议修复顺序

建议按下面顺序处理，而不是四个问题一起平推：

1. 先修复根布局壳层策略。
   *   这是影响面最大的结构性问题。
2. 再移除或收敛未完成模块导航入口。
   *   避免全局 404。
3. 再同步设计文档状态段。
   *   让文档重新和当前分支对齐。
4. 最后修复周视图的时区问题。
   *   这是局部逻辑问题，但仍应尽早处理。

## 4. 对本轮改动的正向评价

虽然上面列了问题，但这轮改动也有几件事是值得保留的：

*   已开始把品牌、壳层和中层展示块从旧单页结构里抽出来。
*   `blocks` 目录的方向是对的，说明你在为后续首页、个人中心、就业广场复用做准备。
*   `AppSidebar` 被明确标注为 legacy/fallback，也符合渐进式迁移思路。
*   设计文档已经开始承担“实现规划 + 协作规则”角色，这对后续接手很有帮助。

所以这次不是“方向错了”，而是：

*   壳层落点还需要再收一次
*   全局入口要比局部组件更保守
*   文档状态要跟代码同步

---

## 5. 建议结论

本轮改动建议状态：

*   **不建议直接作为后续全面铺开的稳定基线**

更准确的处理方式是：

*   先按本审查文档修复 `P1`
*   再同步文档状态
*   然后再继续让首页、个人中心等页面逐步切入新壳层

这样后面继续推进 `/career`、`/interview`、`/advisor-chat` 时，结构会顺很多，也能避免在错误的全局壳层上继续叠代码。
